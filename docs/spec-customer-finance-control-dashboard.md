# אפיון: מסך בקרה כספי ללקוח בפורטל

> סטטוס: אפיון ליישום עתידי תחת `V2-005C`.
> בעלות ראשית: Customer Portal.
> מודולים משתתפים: Customers, Portal, Orders, Finance, Documents, Reports.

## מטרה

פורטל הלקוח צריך להיות מערכת שליטה אמיתית ללקוח, לא רק מסך פתיחת הזמנה.
מנהל הכספים של הלקוח צריך לראות בזמן אמת כמה הוזמן, כמה סופק, כמה חויב, מה עומד לתשלום, ומה מצב התקציב בכל אתר או פרויקט.

המסך צריך לענות על שלוש שאלות:

1. כמה ברזל וכסף כבר יצא על כל אתר?
2. האם יש תשלום שצריך לטפל בו עכשיו לפי תנאי התשלום?
3. מאילו הזמנות אפשר להפיק מסמכים ודוחות בהמשך?

## עקרונות

- הלקוח מגדיר את האתרים והפרויקטים שלו, בתוך ההרשאות שטנא נתנה לו.
- מנהל כספים יכול לראות כסף, תקציבים, חשבוניות ותשלומים.
- מנהל עבודה באתר רואה רק את האתר שלו, ולא רואה מחירים אלא אם קיבל הרשאה מפורשת.
- מי שמשויך לאתר אחד לא בוחר אתר בעת פתיחת הזמנה; ההזמנה נפתחת ישר על האתר שלו.
- מי שמשויך לכמה אתרים בוחר רק מתוך האתרים המורשים לו.
- אין להסיק הרשאות ממעמד כללי כמו `canApprove`; צריך הרשאות מפורשות לכל פעולה.
- כל סכום כסף שמוצג בפורטל חייב להגיע מהשרת אחרי בדיקת הרשאה.

## בעלי תפקידים

| תפקיד | מה הוא רואה | מה הוא יכול לעשות |
|---|---|---|
| מנהל לקוח | כל האתרים, משתמשים, הזמנות והרשאות לפי מה שטנא פתחה לו | לפתוח אתרים, לשייך משתמשים, להגדיר הרשאות פנימיות |
| מנהל כספים | סכומים, תקציבים, חשבוניות, תשלומים צפויים והיסטוריית הזמנות | לסנן לפי אתר, לבדוק חריגות, להפיק דוחות בעתיד |
| מנהל עבודה באתר | הזמנות ומשקלים של האתר שלו | לפתוח הזמנות לאתר שלו בלבד |
| מזמין | פתיחת הזמנה ומעקב בסיסי | ליצור הזמנה לפי ההרשאות שקיבל |

## מסך הבית ללקוח

### ללא אתרים

- אם למשתמש יש `canCreateSites`, מוצג מסך פתיחת אתר ראשון.
- אם אין לו `canCreateSites`, מוצגת הודעה שאין לו אתר משויך ושצריך לפנות למנהל הלקוח.

### אתר אחד

- האתר נבחר אוטומטית.
- אין שדה בחירת אתר בהזמנה.
- מוצגים נתוני האתר, הזמנות אחרונות ופעולות זמינות.

### כמה אתרים

- מוצגת רשימת אתרים מורשים.
- פתיחת הזמנה מחייבת בחירת אתר מתוך הרשימה.
- מנהל כספים יכול לסנן את הדשבורד לפי אתר או לראות הכל יחד.

## דשבורד מנהל כספים

הדשבורד הוא אזור ויזואלי נפרד בפורטל, זמין רק למי שיש לו הרשאות כספיות.

### כרטיסי עליונים

| כרטיס | תוכן |
|---|---|
| לתשלום עכשיו | סכום שהגיע מועד התשלום שלו לפי תנאי התשלום |
| לתשלום בקרוב | סכום שמועדו בתוך 7/30 יום |
| חשיפה פתוחה | הזמנות מאושרות/מסופקות שעדיין לא נסגרו בחשבונית או תשלום |
| ניצול תקציב | אחוז ניצול תקציב לפי האתר הנבחר |
| אתרים בחריגה | מספר אתרים שעברו תקציב כסף או משקל |

### טבלת אתרים

לכל אתר מוצגים:

- שם אתר.
- עיר וכתובת.
- מנהל עבודה אחראי.
- תקציב כסף.
- תקציב טון/ק"ג אם הוגדר.
- הוזמן בק"ג ובכסף.
- אושר בק"ג ובכסף.
- סופק בק"ג ובכסף.
- חויב בחשבוניות.
- נותר לתקציב.
- סטטוס חריגה.

אם למשתמש אין `canViewPrices`, מוצגים רק משקלים וסטטוסים, בלי סכומים.

### התראות תשלום

כאשר מגיע זמן תשלום לפי תנאי התשלום, מוצגת התראה בולטת:

- סכום לתשלום.
- מקור הסכום: חשבונית / תעודת משלוח / הזמנה.
- אתר או כמה אתרים.
- תאריך בסיס לחישוב.
- מועד תשלום.
- כמה ימים עברו או נשארו.
- כפתור צפייה בפירוט.

ההתראה לא מבצעת גבייה בעצמה. היא מציפה למנהל הכספים מה דורש טיפול.

## חישוב מועד תשלום

תנאי התשלום צריכים לכלול עוגן:

| עוגן | משמעות |
|---|---|
| `invoice_date` | המועד נספר מתאריך חשבונית |
| `delivery_note_date` | המועד נספר מתאריך תעודת משלוח |
| `order_approval_date` | המועד נספר מתאריך אישור הזמנה |
| `manual_due_date` | מועד ידני שהוגדר במשרד |

חישוב בסיסי:

```text
due_date = anchor_date + net_days
```

מצבי תשלום:

```text
not_due -> due_soon -> due_now -> overdue -> paid
```

כל חישוב חייב להישמר עם מקור ברור כדי שמנהל הכספים יבין למה קפצה התראה.

## היסטוריית הזמנות

הלקוח צריך לראות היסטוריית הזמנות מלאה, עם סינון לפי:

- אתר.
- טווח תאריכים.
- סטטוס הזמנה.
- סטטוס אספקה.
- סטטוס חשבונית.
- סטטוס תשלום.

כל שורת הזמנה מציגה:

- מספר הזמנה.
- אתר.
- תאריך פתיחה.
- תאריך אספקה אם קיים.
- משקל.
- סכום אם מותר למשתמש לראות מחיר.
- סטטוס.
- מסמכים קיימים.
- פעולות עתידיות להפקת מסמכים.

## מסמכים עתידיים להפקה

בשלב ראשון הכפתורים יכולים להיות מושבתים או מסומנים כעתידי, אבל החוזה צריך להכין אותם:

- סיכום הזמנה.
- תעודת משלוח.
- חשבונית.
- דוח חודשי.
- דוח אתר.
- דוח חריגה מתקציב.

הפקת מסמך בפועל שייכת למודול Documents, אבל פורטל הלקוח מציג את הפעולה.

## הרשאות נדרשות

| הרשאה | משמעות |
|---|---|
| `canCreateSites` | פתיחת אתרים חדשים |
| `canManageUsers` | ניהול משתמשי לקוח |
| `canAssignUsersToSites` | שיוך משתמשים לאתרים |
| `canCreateOrders` | פתיחת הזמנות |
| `canApproveOrders` | אישור הזמנות בתוך הלקוח |
| `canViewPrices` | צפייה במחירים וסכומים |
| `canViewBudget` | צפייה בתקציב |
| `canSetBudget` | הגדרת תקציב אתר |
| `canViewInvoices` | צפייה בחשבוניות |
| `canViewDeliveryNotes` | צפייה בתעודות משלוח |
| `canViewPaymentAlerts` | צפייה בהתראות תשלום |
| `canExportCustomerDocs` | הפקת מסמכים עתידית |

אסור ש-`canApprove` כללי יפתח יכולת פתיחת אתר, צפייה במחיר או ניהול משתמשים.

## API מוצע

| Method | Path | שימוש |
|---|---|---|
| GET | `/api/c/finance/summary` | כרטיסי דשבורד לפי לקוח/אתר/תאריך |
| GET | `/api/c/finance/sites` | טבלת אתרים עם משקל, סכומים ותקציב לפי הרשאה |
| GET | `/api/c/finance/payments-due` | התראות תשלום לפי תנאי תשלום |
| GET | `/api/c/orders/history` | היסטוריית הזמנות מסוננת |
| GET | `/api/c/orders/:id/documents` | מסמכים קיימים להזמנה |
| POST | `/api/c/documents/generate` | עתידי: בקשת הפקת מסמך |

כל endpoint חייב:

- לזהות את הלקוח לפי הטוקן ולא לפי פרמטר חופשי.
- לסנן אתרים לפי הרשאות המשתמש.
- להסיר שדות כסף אם אין `canViewPrices`.
- לרשום פעולה רגישה ביומן ביקורת.

## Definition of Done

- למנהל כספים יש מסך בקרה ויזואלי נפרד בפורטל.
- ניתן לראות סכומים לתשלום עכשיו ולתשלום בקרוב לפי תנאי תשלום.
- ניתן לפצל את כל המדדים לפי אתר.
- מנהל עבודה באתר יחיד לא נדרש לבחור אתר.
- משתמש ללא הרשאת מחיר לא מקבל סכומי כסף מהשרת.
- קיימת היסטוריית הזמנות עם מקום לפעולות מסמכים עתידיות.
- קיימות בדיקות שמוודאות הפרדת הרשאות בין מנהל כספים למנהל עבודה.
- אין שימוש ב-`canApprove` כקיצור דרך להרשאות אתר, מחיר או כספים.

## שאלות פתוחות למאיר

- האם דוח חודשי צריך להיות לפי אתר, לפי לקוח, או שניהם?
- האם מנהל כספים יכול לאשר חריגה מתקציב או שרק טנא מאשרת?
