# תקן רספונסיביות IronBend

Date: 2026-07-05
Owner: UX/UI
Scope: תקן ומפת עבודה בלבד. אין שינוי קוד, API, DB, Orders, Production, Finance, OCR או Shape V2 Contract במסמך הזה.

## למה הבעיה חוזרת

המערכת לא נשברת בגלל מסך אחד. הבעיה היא שאין שכבת רספונסיביות אחת שמחייבת את כל המסכים.

כרגע קיימים הרבה פתרונות מקומיים: רוחבים קבועים, פאנלים ב־px, טבלאות רחבות, כרטיסיות הדפסה במידות קשיחות, מודאלים שונים, ו־CSS מוטמע בתוך כל מסך. לכן כל פעם שמתקנים מסך אחד, אותה בעיה חוזרת במסך אחר.

המטרה היא לעבור מתיקון נקודתי לתקן מערכת.

## עקרונות מחייבים

1. מסך לא מקבל פתרון רספונסיבי פרטי אם יש רכיב מערכת מתאים.
2. כל מסך חייב לעבוד בארבעה מצבים: מובייל, טאבלט, דסקטופ, מסך רחב.
3. טבלאות רחבות לא שוברות מסך. הן מקבלות גלילה אופקית או תצוגת כרטיסים לפי המסך.
4. פאנל צד לא מכסה תוכן. במסכים צרים הוא הופך לאזור תחתון, drawer, או נפתח לפי פעולה.
5. הדפסה היא תקן נפרד ממסך. לא מתקנים הדפסה דרך CSS של תצוגת עבודה.
6. לא משתמשים ברוחב קשיח ללא `max-width`, `minmax`, `overflow`, או התנהגות מובייל מוגדרת.
7. שינוי UI לא משנה DOM עסקי, IDs, endpoints, events, חישובים או חוזים.

## Breakpoints מחייבים

| מצב | רוחב בדיקה | כלל UI |
|---|---:|---|
| Mobile narrow | 390px | עמודה אחת, פעולות תחתונות, טבלאות בגלילה או כרטיסים |
| Mobile wide | 430px | שדות גדולים, ללא חיתוך טקסט, פאנלים לא חופפים |
| Tablet | 768px | שתי עמודות רק אם נשאר מקום קריא |
| Desktop | 1366px | פריסת עבודה רגילה עם nav/panels |
| Wide desktop | 1920px | תוכן לא נמרח בלי גבול; אזורי עבודה מקבלים max-width חכם |
| Print | A4 | מידות פיזיות, ללא תלות ברוחב הדפדפן |

## שכבת Layout אחידה

כל מסך מערכת צריך להתכנס לאחד מדפוסי layout הבאים:

| Layout | שימוש | כלל |
|---|---|---|
| App shell | מסכי ניהול רגילים | ניווט צד, כותרת, תוכן מרכזי |
| Workbench | הזמנה חדשה, OCR, Shape Editor | אזור עבודה מרכזי, פאנל פרמטרים, פעולות קבועות |
| Document | מחירון, הדפסות, מסמכים | דף/מסמך במרכז, toolbar מינימלי |
| Operator | עובד/ייצור/קופה/קישור סריקה | כפתורים גדולים, מינימום טקסט, מובייל קודם |
| Portal | פורטל לקוח | זרימה פשוטה, כרטיסים ברורים, ללא חשיפת שדות פנימיים |

## רכיבי בסיס שצריך לאחד

| רכיב | מצב נוכחי | יעד |
|---|---|---|

| Forms | שדות וריווחים שונים | field row אחיד עם label, unit, hint |
| Tables | חלק רחבות מדי | wrapper אחיד עם גלילה או cards במובייל |
| Modals | גדלים והתנהגות שונים | modal shell אחיד, בלי שינוי logic |
| Tabs | סגנונות שונים | tabs אחידים למסכי מערכת |
| KPI | כרטיסי מדדים כפולים | KPI card אחיד |
| Empty states | מסכים ריקים שונים | מצב ריק/שגיאה/טעינה אחיד |

## כללי ביצוע לכל Slice

כל Slice רספונסיבי חייב לכלול:

- scope של מסך אחד או רכיב אחד בלבד.
- רשימת קבצים מדויקת לפני עבודה.
- ללא שינוי API/DB/חישובים/סטטוסים.
- צילום לפני/אחרי בדסקטופ ובמובייל אם אפשר.
- בדיקה ב־390, 768, 1366, 1920.
- בדיקה שאין overlap ואין טקסט חתוך.
- `npm test` אם נגעו בקבצים שמשפיעים על חוזים או renderers.
- commit עם pathspec מפורש.

## Do Not Touch ללא משימה נפרדת

- `routes/`, `db/`, `services/` לוגיקה עסקית.
- Shape V2 Contract, חישובי אורך/משקל/זוויות.
- Orders lifecycle, Production statuses, Finance/Pricing calculations.
- OCR parsing והשוואת מקור.
- הרשאות, JWT, portal tokens.
- מידות הדפסה פיזיות של כרטיסיות בלי אישור מפורש.

## החלטת עבודה

מכאן והלאה, כל בקשת UX/UI שיש לה השפעה על layout תיבדק מול המסמך הזה.

אם שינוי דורש חריגה מהתקן, צריך לכתוב למה החריגה נדרשת ומה ההשפעה שלה על מובייל, דסקטופ והדפסה.
